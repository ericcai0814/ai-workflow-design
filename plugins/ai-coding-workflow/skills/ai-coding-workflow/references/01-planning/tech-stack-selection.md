# 技術棧選擇指南

## 概述

技術棧選擇影響專案的長期維護性和開發效率。本文件提供選擇技術棧的決策框架和與 AI 協作的 Prompt。

## 前置條件

- [ ] 已完成需求分析
- [ ] 了解團隊技能組成
- [ ] 了解專案限制（預算、時程、規模）

## 決策框架

### 考量因素

| 因素 | 權重 | 說明 |
|------|------|------|
| 團隊熟悉度 | 高 | 團隊是否有經驗 |
| 生態系統 | 高 | 套件、工具、社群支援 |
| 長期維護 | 高 | 活躍度、穩定性、向後相容 |
| 效能需求 | 中 | 是否滿足效能要求 |
| 學習曲線 | 中 | 新成員上手難度 |
| 招募市場 | 中 | 是否容易找到人才 |

### 常見技術棧組合

#### 前端

| 組合 | 適用場景 |
|------|----------|
| React + Next.js + TypeScript | 大型專案、SEO 需求、團隊熟悉 React |
| Vue + Nuxt + TypeScript | 中小型專案、快速開發、漸進式採用 |
| Astro + React/Vue | 內容網站、極致效能、靜態為主 |
| SvelteKit | 追求效能、願意嘗試新技術 |

#### 後端

| 組合 | 適用場景 |
|------|----------|
| Node.js + Express/Fastify | 全棧 JS、即時應用、快速原型 |
| Node.js + NestJS | 大型專案、企業級、需要結構化 |
| Python + FastAPI | ML/AI 整合、資料處理 |
| Go + Gin/Fiber | 高併發、微服務、效能敏感 |

#### 資料庫

| 類型 | 選項 | 適用場景 |
|------|------|----------|
| 關聯式 | PostgreSQL | 通用、複雜查詢、ACID |
| 關聯式 | MySQL | 讀取密集、成熟穩定 |
| 文件式 | MongoDB | 彈性 Schema、快速迭代 |
| Serverless | Neon, PlanetScale | 無伺服器部署、自動擴展 |

## 常見錯誤

### 1. 追新不追穩

**問題**：選擇最新最潮的技術，但生態系統不成熟。

**建議**：選擇「無聊」但穩定的技術，除非有明確的理由需要新技術。

### 2. 過度設計

**問題**：小專案使用微服務、Kubernetes 等複雜架構。

**建議**：從單體開始，需要時再拆分。

### 3. 忽略團隊因素

**問題**：選擇「最好」的技術，但團隊沒人會用。

**建議**：技術能力 × 團隊熟悉度 = 實際產出。
