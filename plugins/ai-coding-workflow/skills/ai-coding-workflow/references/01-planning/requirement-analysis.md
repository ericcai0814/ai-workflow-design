# 需求分析流程

## 概述

需求分析是開發的第一步，目標是將模糊的需求轉換為可執行的任務。本文件提供與 AI 協作進行需求分析的流程和 Prompt。

## 前置條件

- [ ] 有初步的需求描述（口頭、文件、或 issue）
- [ ] 了解專案背景和現有功能
- [ ] 確定利害關係人

## 工作流程

### 步驟 1：收集原始需求

記錄所有相關資訊：
- 用戶/客戶的原始描述
- 相關的 issue 或討論
- 競品參考
- 設計稿（如有）

### 步驟 2：釐清需求

與 AI 協作，將模糊需求轉換為具體問題：

1. 識別需求中的模糊點
2. 列出需要確認的問題
3. 定義範圍邊界（做什麼、不做什麼）

### 步驟 3：拆解用戶故事

將需求拆解為用戶故事（User Story）格式：

```
作為 [角色]，
我想要 [功能]，
以便 [價值/目的]。
```

### 步驟 4：定義驗收標準

為每個用戶故事定義可驗證的驗收標準（Acceptance Criteria）。

### 步驟 5：評估複雜度

評估技術複雜度和工作量，識別風險點。

## Prompts

### Prompt 1：需求釐清與問題識別

```
你是一位資深產品經理和需求分析師。請分析以下需求，識別模糊點並提出問題。

## 原始需求
${RAW_REQUIREMENT}

## 專案背景
${PROJECT_CONTEXT}

## 請執行以下分析

### 1. 需求理解
用你自己的話重述這個需求，確認理解正確。

### 2. 模糊點識別
列出需求中不夠明確的地方：
- 功能範圍模糊
- 用戶角色不清
- 邊界條件未定義
- 技術限制未說明

### 3. 待確認問題
列出需要與利害關係人確認的問題（按優先級排序）：
1. [必須確認] ...
2. [建議確認] ...
3. [可選確認] ...

### 4. 假設列表
如果某些問題無法立即確認，列出你會做的假設。

### 5. 範圍建議
建議這個需求的範圍邊界：
- 包含：...
- 不包含：...
- 未來可擴展：...
```

### Prompt 2：用戶故事拆解

```
你是一位敏捷教練。請將以下需求拆解為用戶故事。

## 需求描述
${REQUIREMENT_DESCRIPTION}

## 已知的用戶角色
${USER_ROLES}

## 請輸出

### 用戶故事列表

對每個故事，請提供：

#### US-1: [故事標題]

**故事描述**
作為 [角色]，
我想要 [功能]，
以便 [價值]。

**驗收標準**
- [ ] Given [前置條件], When [動作], Then [結果]
- [ ] Given ..., When ..., Then ...

**技術備註**
- 相關的技術考量
- 可能的實作方式

**依賴關係**
- 依賴 US-X
- 被 US-Y 依賴

---

### 優先級建議

| 優先級 | 故事 | 原因 |
|--------|------|------|
| P0 | US-X | ... |
| P1 | US-Y | ... |
| P2 | US-Z | ... |
```

推薦方案 [A/B]，原因：...

在開始實作前需要：
- [ ] ...

## 檢查清單

### 需求完整性

- [ ] 需求描述清晰，無重大模糊點
- [ ] 已識別所有用戶角色
- [ ] 已定義功能範圍邊界
- [ ] 已列出待確認問題並獲得答案

### 用戶故事品質

- [ ] 每個故事都有明確的角色、功能、價值
- [ ] 每個故事都有可驗證的驗收標準
- [ ] 故事大小適中（可在一個 sprint 完成）
- [ ] 依賴關係已識別

### 技術評估

- [ ] 已評估技術可行性
- [ ] 已識別潛在風險
- [ ] 已估計工作量
- [ ] 已確定技術方案

## 常見問題

### Q1：需求太模糊怎麼辦？

**A1**：使用 Prompt 1 識別模糊點，列出問題清單與利害關係人確認。如果無法立即確認，先列出假設繼續推進，但標記為待確認。

### Q2：用戶故事應該多大？

**A2**：理想的用戶故事應該：
- 可在 1-3 天內完成
- 獨立可交付價值
- 可獨立測試

如果太大，繼續拆解；如果太小，考慮合併。

### Q3：如何處理技術驅動的需求？

**A3**：即使是技術需求（如效能優化、重構），也建議用用戶故事格式描述：

```
作為開發者，
我想要將 API 響應時間降到 200ms 以下，
以便提升用戶體驗和 SEO 排名。
```
