# Prompt: 需求分析

## 任務目標

將模糊的需求描述轉化為清晰、可執行的功能規格。

## 使用時機

- 收到新功能需求時
- 需求描述不夠清晰時
- 需要確認功能範圍時

---

## 變數說明

執行此 prompt 前，請先取得以下值：

| 變數                      | 來源                 | 說明                         | 範例                                                       |
| ------------------------- | -------------------- | ---------------------------- | ---------------------------------------------------------- |
| `${ORIGINAL_REQUIREMENT}` | 用戶本次對話         | 用戶描述的原始需求，原文引用 | "我想要一個可以拖放排序的待辦清單"                         |
| `${PROJECT_CONTEXT}`      | detect-context agent | 專案技術棧、狀態、結構的摘要 | "Next.js 14 + TypeScript + Tailwind CSS，已有基本頁面結構" |

**執行時，將所有 `${...}` 替換為實際值。如果 detect-context 無法偵測，請詢問用戶。**

---

## Prompt 範本

```prompt
你是一位資深產品經理。請分析以下需求並產出結構化的規格。

## 原始需求
${ORIGINAL_REQUIREMENT}

## 專案背景
${PROJECT_CONTEXT}

## 請分析並輸出

### 1. 需求理解確認

我理解你想要：
- [用一句話描述核心需求]

### 2. 功能清單

#### 必要功能 (Must Have)
- [ ] 功能 1：[描述]
- [ ] 功能 2：[描述]

#### 選擇功能 (Nice to Have)
- [ ] 功能 3：[描述]

#### 不包含 (Out of Scope)
- [明確列出不包含的功能]

### 3. 使用者故事

作為 [角色]，我想要 [功能]，以便 [價值]。

### 4. 驗收標準

- [ ] 標準 1
- [ ] 標準 2
- [ ] 標準 3

### 5. 待釐清問題

| 問題 | 影響 | 建議 |
|------|------|------|
| [問題描述] | [如果不釐清會有什麼影響] | [建議的解決方向] |

### 6. 技術考量

- 前端影響：[簡述]
- 後端影響：[簡述]
- 資料庫影響：[簡述]
```

## 執行後驗證

- [ ] 核心需求已清楚描述
- [ ] 功能清單已分類（Must Have / Nice to Have）
- [ ] 有明確的驗收標準
- [ ] 待釐清問題已列出

## 下一步

- 確認待釐清問題 → 與需求方討論
- 功能清單確認後 → 執行 `create-plan.md`

## 常見問題

**Q: 需求方無法回答待釐清問題怎麼辦？**

A: 先記錄假設，標註「待確認」，繼續進行。在開發過程中持續確認。

**Q: 功能範圍太大怎麼辦？**

A: 建議分階段交付，先完成 Must Have，後續迭代加入 Nice to Have。
